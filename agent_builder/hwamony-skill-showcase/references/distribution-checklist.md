# Distribution Checklist

Use this reference when the user wants the skill or repository to be easier to discover, share, or submit publicly.

## GitHub Surface Checklist

- GitHub "About" sentence is short, concrete, and aligned with the README promise
- repository description is concise and concrete
- repository topics reflect the actual audience and ecosystem
- homepage or docs link supports the README claim
- badges or credibility signals help the visitor orient quickly
- social preview image exists
- README opening explains value before structure
- one quickstart path exists
- at least one flagship example is visible
- release notes are understandable without opening the diff

## Proof Checklist

- example prompts exist
- expected outcomes are shown
- at least one realistic demo path exists
- claims match actual repo contents

## Submission Checklist

For awesome lists or directories:

- one-sentence blurb
- repo URL
- category fit
- what makes it distinct
- evidence of maintenance or real use, if available

## Messaging Checklist

- can a new visitor explain what this is after 10 seconds?
- is there a memorable line or handle for the project?
- is the target user visible?
- is the promise concrete?
- is there a believable next step?
- is the copy specific without sounding defensive?

## Priority Rule

If time is short, improve in this order:

1. README first screen
2. example prompts
3. GitHub description and topics
4. release copy
5. broader distribution copy
