# README Structure

Use this reference when rewriting a repo README, category README, or public intro section.

## Default Root README Order

1. Title
2. One-line value proposition
3. Short explanation of the collection
4. Flagship skills or categories
5. Quickstart
6. Example prompts or use cases
7. Why this repo is different
8. Contribution or extension path
9. Licensing

## Default Category README Order

1. Category name
2. What this category helps with
3. Included skills with one-line differences
4. Who it is for
5. Suggested starting path
6. License note

## Default Single-Skill Showcase Order

1. Skill name
2. One-line promise
3. When to use it
4. What it produces
5. Example prompts
6. Notes or boundaries

## Heuristics

- Put the "why care" section before the inventory.
- Put the "try this" section before the long explanation.
- Move licensing and attribution below the user-value sections unless legal constraints require otherwise.
- If the page feels like a directory listing, add a stronger point of view.
- If the page feels hype-heavy, reduce claims and add proof.

## Red Flags

- The first screen contains only structure, not value.
- Readers need to open multiple files before understanding the point.
- Every skill is described with the same generic phrasing.
- The page says what is included, but not what outcome the user gets.
